## 2. Micro-Decision Resolutions

These are implementation-level decisions that the normative specs deliberately
left open. Each is resolved here to eliminate ambiguity for the implementing
engineer.

### MD-1: Supervisor-to-Backend Authentication

**Decision:** The supervisor authenticates to the backend using an **API key**
via the existing `dualAuthProcedure` / `apiKeyProcedure` mechanism.

**Rationale:**
- The supervisor is a headless process, not a browser session. API key auth
  is the natural fit (same as SDK ingest).
- The existing `apiKeyProcedure` in `packages/backend/src/trpc/trpc.ts`
  (L104-136) already handles Bearer token auth, SHA-256 key lookup, tenant
  scoping, and RLS context setting.
- The supervisor process receives its API key via the `LOOPSTORM_API_KEY`
  environment variable. The key is created by the operator in the dashboard
  with `["ingest", "read", "supervisor"]` scopes.

**New scope:** Add `"supervisor"` to the allowed scopes list. The
`supervisor-tools.*` procedures check for this scope. This prevents regular
SDK keys (with only `["ingest"]`) from accessing supervisor tool APIs.

**No new auth middleware needed.** The supervisor package communicates with
the backend over HTTP using a standard tRPC client with `Authorization: Bearer <key>`.

### MD-2: Supervisor Tool API -- New Router vs. Extended Router

**Decision:** Create a new `supervisorToolsRouter` at
`packages/backend/src/trpc/routers/supervisor-tools.ts`. Do NOT extend the
existing `supervisorRouter` (which handles human-facing proposal/escalation
management).

**Rationale:** The existing `supervisorRouter` serves dashboard users
(protectedProcedure = session auth). The new `supervisorToolsRouter` serves
the supervisor process (apiKeyProcedure = API key auth with `supervisor`
scope). Mixing auth strategies in one router is confusing. The two routers
serve different consumers with different auth requirements.

**Mounting:** Add to `packages/backend/src/trpc/router.ts`:
```typescript
supervisorTools: supervisorToolsRouter,
```

### MD-3: Mock LLM Strategy for Tests

**Decision:** All supervisor tests use a **deterministic mock `LLMProvider`**
that returns pre-scripted tool calls. No real LLM API calls in any test
(unit, integration, or E2E).

**Implementation:**
```typescript
// apps/supervisor/src/llm/mock.ts
class MockLLMProvider implements LLMProvider {
  private responses: LLMResponse[];
  private callIndex = 0;

  constructor(responses: LLMResponse[]) {
    this.responses = responses;
  }

  async chat(_params: ChatParams): Promise<LLMResponse> {
    const response = this.responses[this.callIndex];
    if (!response) {
      return {
        content: [{ type: "text", text: "Session complete." }],
        stop_reason: "end_turn",
        usage: { input_tokens: 0, output_tokens: 0 },
      };
    }
    this.callIndex++;
    return response;
  }
}
```

**Test fixture pattern:** Each test scenario provides a sequence of
`LLMResponse` objects that script the tool calls the supervisor will make.
The tool implementations execute against a real (test) database or mocked
tRPC responses. This tests the full session lifecycle without LLM
variability.

**CI requirement:** The `test-supervisor` CI job must NOT require an
`ANTHROPIC_API_KEY` environment variable. Tests that would need a real key
must use the mock provider.

### MD-4: Trigger Dispatch -- Synchronous vs. Asynchronous

**Decision:** Trigger evaluation is **synchronous** (inline in event ingest).
Trigger dispatch is **asynchronous** (enqueue to bounded channel, background
worker processes).

**Rationale:**
- Trigger evaluation is fast: a few comparisons against the run's aggregate
  fields. It adds < 1ms to the ingest hot path.
- Trigger dispatch (starting a supervisor session) is slow: it involves an
  HTTP call to the supervisor process. This MUST NOT block event ingest.
- ADR-014 Gate 3 resolved this: in-process bounded async channel, cap 100.

**Implementation:**
- `evaluateTriggers()` is called at the end of the `events.ingest`
  transaction (after all DB writes succeed).
- If triggers fire, a `TriggerMessage` is enqueued to a `TriggerQueue`
  singleton (bounded channel, cap 100).
- A background worker reads from the queue and dispatches HTTP requests
  to the supervisor process endpoint.
- If the queue is full, new triggers are dropped with a warning log.

### MD-5: Supervisor HTTP Endpoint

**Decision:** The supervisor process exposes a single HTTP endpoint for
trigger dispatch:

```
POST /api/trigger
Content-Type: application/json
Authorization: Bearer <supervisor-internal-key>

{
  "trigger": "terminated_budget",
  "trigger_run_id": "019606f0-...",
  "tenant_id": "a1b2c3...",
  "priority": 1,
  "backend_url": "http://localhost:3001",
  "api_key": "lsg_..."
}
```

The supervisor process is a long-running Bun HTTP server (not a one-shot
script). It receives trigger messages, creates a `SupervisorSession` for
each, and runs them to completion.

**Internal auth:** The trigger dispatch worker in the backend authenticates
to the supervisor process using a shared secret
(`LOOPSTORM_SUPERVISOR_INTERNAL_KEY`). This is a simple Bearer token check,
not the full API key mechanism.

### MD-6: Proposal and Escalation ID Generation

**Decision:** The supervisor generates `proposal_id` and `escalation_id`
values using the format defined in the spec:
- `proposal_id`: `"prop_"` + 8 random hex chars (e.g., `"prop_a1b2c3d4"`)
- `escalation_id`: `"esc_"` + 8 random hex chars (e.g., `"esc_e5f6a7b8"`)

These are generated by the supervisor process (not the backend). The backend
stores them as-is in the `proposal_id` / `escalation_id` columns.

### MD-7: Learning Store Write Path

**Decision:** Learning tools (`record_incident_pattern`, `update_agent_profile`,
`record_intervention_outcome`) write to the `supervisor_proposals` table with
a synthetic `proposal_type` value:

| Tool | `proposal_type` value | `status` |
|---|---|---|
| `record_incident_pattern` | `"incident_pattern"` | `"approved"` (auto) |
| `update_agent_profile` | `"agent_profile_update"` | `"approved"` (auto) |
| `record_intervention_outcome` | `"intervention_outcome"` | `"approved"` (auto) |

The data is stored in the `proposed_changes` JSONB column. These records
enter as `"approved"` immediately (they are observations, not change
requests -- no human approval needed).

This is the v1.1 approach per spec Section 4.5. A dedicated learning store
table is a v2 consideration.

### MD-8: `runs.getEvents` BT Field Gap

**Decision:** Add the 4 BT columns to the `runs.getEvents` select list.
This is a one-line-per-field addition (4 lines total). Without this, the
supervisor's `read_run_events` tool would receive events without BT fields,
breaking the `compute_risk_score` signals S5-S8.

### MD-9: Branch Strategy

**Decision:** Single feature branch `feat/v1.1-ai-supervisor`, with the
following PR structure:

| PR | Scope | Why separate |
|---|---|---|
| PR-A | Backend extensions (SUP-A: triggers, tool APIs, timeout job, resolve escalation, getEvents BT fix) | Can be reviewed and merged first, unblocking frontend work. |
| PR-B | `apps/supervisor/` package + `examples/supervisor-policy.yaml` | Core feature. Depends on PR-A. |
| PR-C | Frontend extensions (resolve escalation, activity log) | Small delta on existing pages. Can be parallel with PR-B. |

If the team prefers a single PR, that is acceptable. The sub-phase

#### Task SUP-A3: Trigger Evaluation Logic (SUP-G2)

**Location:** `packages/backend/src/lib/triggers.ts` (new file)

**Exports:**
```typescript
interface TriggerResult {
  trigger: string;
  priority: number;  // 1=critical, 2=high, 3=medium, 4=low
}

function evaluatePostRunTriggers(
  run: { status: string; total_cost_usd: number },
  events: Array<{ event_type: string; decision: string | null }>,
  budgetHardCap: number | null,
): TriggerResult[];

function evaluateMidRunTriggers(
  run: { status: string; total_cost_usd: number; total_call_count: number },
  newEvent: { event_type: string; decision: string | null; rule_id: string | null },
  runningCounters: RunningCounters,
  agentBaseline: { avg_deny_rate: number } | null,
  budgetHardCap: number | null,
): TriggerResult[];

interface RunningCounters {
  deny_count: number;
  total_decisions: number;
  rule_id_counts: Map<string, number>;
}
```

**Post-run triggers (5 types):**

| Trigger | Condition | Priority |
|---|---|---|
| `terminated_budget` | `run.status === "terminated_budget"` | 1 |
| `terminated_loop` | `run.status === "terminated_loop"` | 1 |
| `abandoned` | `run.status === "abandoned"` | 3 |
| `deny_decisions` | Any event with `decision === "deny"` | 3 |
| `high_cost` | `run.total_cost_usd > 0.80 * budgetHardCap` | 2 |

**Mid-run triggers (4 types):**

| Trigger | Condition | Priority |
|---|---|---|
| `budget_warning` | `run.total_cost_usd > 0.70 * budgetHardCap` | 2 |
| `loop_detected` | `newEvent.event_type === "loop_detected"` | 2 |
| `deny_spike` | `deny_count / total_decisions > 3 * agentBaseline.avg_deny_rate` AND `deny_count >= 5` | 2 |
| `repeated_rule` | `rule_id_counts.get(newEvent.rule_id) >= 3` | 3 |

**Edge cases:**
- `high_cost` cannot fire if `budgetHardCap` is null (no budget configured).
- `deny_spike` uses absolute threshold of 50% if no baseline available.
- `deny_spike` requires at least 5 deny decisions to avoid noise on small runs.
- `repeated_rule` excludes `__builtin_*` rule_ids (built-in rules fire
  frequently by design).

**Acceptance criteria:**
- AC-A3-1: `evaluatePostRunTriggers` fires on terminated_budget.
- AC-A3-2: `evaluatePostRunTriggers` fires on terminated_loop.
- AC-A3-3: `evaluatePostRunTriggers` fires on abandoned.
- AC-A3-4: `evaluatePostRunTriggers` fires on deny_decisions (any deny present).
- AC-A3-5: `evaluatePostRunTriggers` fires on high_cost when >80% of hard cap.
- AC-A3-6: `evaluatePostRunTriggers` does NOT fire high_cost when budget is null.
- AC-A3-7: `evaluateMidRunTriggers` fires on budget_warning at 70%.
- AC-A3-8: `evaluateMidRunTriggers` fires on loop_detected event.
- AC-A3-9: `evaluateMidRunTriggers` fires on deny_spike with baseline.
- AC-A3-10: `evaluateMidRunTriggers` fires on deny_spike with absolute fallback.
- AC-A3-11: `evaluateMidRunTriggers` fires on repeated_rule (3+ same rule_id).
- AC-A3-12: `evaluateMidRunTriggers` does NOT fire repeated_rule for `__builtin_*` rule_ids.
- AC-A3-13: Priority ordering is correct (1=critical first).
- AC-A3-14: All trigger functions are pure (no DB access, no side effects).

#### Task SUP-A5: Supervisor Tools tRPC Router (SUP-G3)

**Location:** `packages/backend/src/trpc/routers/supervisor-tools.ts` (new file)

This router provides the backend APIs that the supervisor's tools call.
It uses `apiKeyProcedure` (API key auth) and requires the `"supervisor"`
scope.

**Procedures:**

| Procedure | Backs tool | Auth | Description |
|---|---|---|---|
| `supervisorTools.getRunEvents` | `read_run_events` | API key + supervisor scope | Thin wrapper around `runs.getEvents` with supervisor-specific defaults (limit 500, includes BT fields). |
| `supervisorTools.getAgentBaseline` | `read_agent_baseline` | API key + supervisor scope | Aggregate query: run count, avg/median/p95 cost, avg call count, deny rate, top tools, top denied tools. |
| `supervisorTools.getPolicyPack` | `read_policy_pack` | API key + supervisor scope | Thin wrapper around `policies.get`. |
| `supervisorTools.querySimilarRuns` | `query_similar_runs` | API key + supervisor scope | Fingerprint prefix match on `call_seq_fingerprint` in events table. `scope="anonymous_aggregate"` returns empty. |
| `supervisorTools.createProposal` | `propose_budget_adjustment`, `flag_for_review` | API key + supervisor scope | Insert into `supervisor_proposals`. Returns `proposal_id`. |
| `supervisorTools.createEscalation` | `escalate_to_human` | API key + supervisor scope | Insert into `supervisor_escalations`. Returns `escalation_id`. |
| `supervisorTools.recordLearning` | `record_incident_pattern`, `update_agent_profile`, `record_intervention_outcome` | API key + supervisor scope | Insert into `supervisor_proposals` with learning-specific `proposal_type` and `status: "approved"`. |

**`getAgentBaseline` SQL shape:**

```sql
SELECT
  COUNT(*)::int AS run_count,
  AVG(total_cost_usd) AS avg_cost_usd,
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY total_cost_usd) AS median_cost_usd,
  PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY total_cost_usd) AS p95_cost_usd,
  AVG(total_call_count) AS avg_call_count,
  AVG(total_input_tokens) AS avg_input_tokens,
  AVG(total_output_tokens) AS avg_output_tokens
FROM runs
WHERE tenant_id = $1
  AND agent_name = $2
  AND created_at > NOW() - INTERVAL '$3 days'
```

The deny counts, top tools, and top denied tools require a join to the
`events` table:

```sql
SELECT
  COUNT(*) FILTER (WHERE decision = 'deny')::int AS total_deny_count,
  COUNT(*) FILTER (WHERE event_type = 'policy_decision')::int AS total_decision_count
FROM events
WHERE tenant_id = $1
  AND run_id IN (SELECT run_id FROM runs WHERE agent_name = $2 AND ...)
```

Top tools and top denied tools are separate GROUP BY queries.

**Performance note:** `getAgentBaseline` is an expensive query. Cache the
result per `(tenant_id, agent_name)` for 5 minutes using an in-process
`Map` with TTL. This is acceptable because the baseline changes slowly (new
runs are added over hours/days).

#### Task SUP-B9: Session Manager

**Location:** `apps/supervisor/src/session.ts`

The `SupervisorSession` class orchestrates a single supervisor session:

```typescript
class SupervisorSession {
  readonly supervisorRunId: string;    // "sup_" + 8 hex chars
  readonly triggerRunId: string;
  readonly trigger: string;
  readonly tenantId: string;

  constructor(params: SessionParams);
  async execute(): Promise<SessionResult>;
}
```

**Session lifecycle:**
1. Generate `supervisorRunId` (format: `"sup_"` + 8 random hex chars).
2. Create backend client from config.
3. Emit `supervisor_run_started` event (via backend ingest).
4. Enter tool-use loop:
   a. Send system prompt + trigger context to LLM.
   b. LLM returns content blocks (text and/or tool_use).
   c. For each `tool_use` block: look up handler in registry, execute, return result.
   d. Emit `supervisor_tool_call` event for each tool execution.
   e. Send tool results back to LLM.
   f. Repeat until `stop_reason === "end_turn"` or `"max_tokens"`.
5. Emit `supervisor_proposal_created` / `supervisor_escalation_created`
   events as they are created (inline, not at end).
6. Track cumulative token usage against budget.
7. If budget exceeded (tracked client-side, not via engine in v1.1):
   log budget breach and terminate session.
8. Return `SessionResult` with summary.

**Budget tracking (v1.1 simplification):** The supervisor tracks its own
token budget client-side (cumulative `usage.input_tokens` +
`usage.output_tokens`). The $2.00 hard cap is enforced by converting tokens
to cost using the model's pricing. The supervisor does NOT use
`loopstorm.wrap()` in v1.1 -- that requires a separate engine instance
and adds deployment complexity that is not justified for the MVP.

**IMPORTANT DEVIATION FROM SPEC:** The spec (Section 2.4) says the
supervisor runs through `loopstorm.wrap()`. For v1.1 MVP, the supervisor
tracks its own budget and audit trail client-side. The `loopstorm.wrap()`
integration is a v1.2 enhancement. The supervisor still:
- Emits JSONL events via backend ingest (audit trail is intact).
- Tracks budget (client-side, not engine-enforced).
- Has a policy (but it is enforced by application logic, not the engine).

This deviation is acceptable because:
1. The supervisor's audit trail is still complete and verifiable.
2. Budget enforcement is still present (just not engine-enforced).
3. The v1.1 scope is already XL; adding engine instance management adds
   deployment complexity without meaningfully improving safety.
4. The enforcement/observation plane separation is maintained (the
   supervisor still has no access to the customer engine).

**Acceptance criteria:**
- AC-B9-1: Session generates valid `supervisorRunId` format.
- AC-B9-2: Session emits `supervisor_run_started` event at start.
- AC-B9-3: Session correctly executes the tool-use loop.
- AC-B9-4: Session emits `supervisor_tool_call` for each tool execution.
- AC-B9-5: Session stops on `end_turn` stop reason.
- AC-B9-6: Session stops on budget exhaustion.
- AC-B9-7: Session handles LLM errors gracefully (logs, does not crash).
- AC-B9-8: Full session with mock LLM completes end-to-end.

## 5. Invariant Checklist

Every implementation PR must verify these invariants before merge:

- [ ] **I-1:** `escalate_to_human` is callable by the supervisor under any
  conditions. Test: mock LLM calls `escalate_to_human`, verify escalation
  is created.
- [ ] **I-2:** Enforcement/observation plane separation: supervisor has no
  import from `apps/engine/`, no UDS connection to customer engine socket,
  no write to customer JSONL.
- [ ] **I-3:** AGPL/MIT boundary: `apps/supervisor/` (MIT) has no runtime
  import from `packages/backend/` (AGPL) or `packages/web/` (AGPL).
- [ ] **I-4:** Supervisor events are stored with the triggering run's
  `tenant_id` (same tenant, not a system tenant).
- [ ] **I-5:** `supervisor_proposals.status` uses `"pending"` in DB,
  supervisor events use `"pending_approval"` in `event.status`. Backend
  maps between them.
- [ ] **I-6:** Budget cap is enforced (session terminates when cost exceeds
  $2.00 or call count exceeds 100).
- [ ] **I-7:** Supervisor cannot read other tenants' data (RLS enforced).
- [ ] **I-8:** No `ANTHROPIC_API_KEY` required in CI tests.

## 6. Risk Areas

| ID | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | `getAgentBaseline` query is expensive on large tenants | Medium | 5-minute in-process cache with TTL. |
| R2 | `@anthropic-ai/sdk` API may change before v1.1 ships | Low | Pin to `^0.36.0`. Mock provider isolates tests from SDK changes. |
| R3 | Trigger dispatch adds latency to event ingest | Medium | Fire-and-forget: trigger eval is inline (fast), dispatch is async. |
| R4 | Supervisor sessions may hang if LLM returns unexpected shapes | Medium | Timeout per LLM call (30s). Catch all errors in session loop. |
| R5 | Status mapping (pending vs. pending_approval) causes confusion | Low | Document in code comments. Test both paths. |
| R6 | v1.1 MVP does not use `loopstorm.wrap()` for self-guarding | Medium | Client-side budget tracking + audit trail via backend ingest. Document as v1.2 enhancement. |
| R7 | Concurrent supervisor sessions may race on proposal dedup | Low | Proposal dedup by `(target_id, dimension)` uses DB unique constraint. |
| R8 | AGPL/MIT boundary violation if engineer imports backend module | High | CI `license-boundary` check. Code review. |

## 8. Deviation Log

Deviations from normative specs, with rationale:

| ID | Spec section | Deviation | Rationale |
|---|---|---|---|
| DEV-1 | ai-supervisory-interface.md Section 2.4 | v1.1 supervisor does NOT use `loopstorm.wrap()` for self-guarding. Uses client-side budget tracking + backend ingest for audit trail. | `loopstorm.wrap()` requires deploying and managing a separate engine instance. This adds significant deployment complexity for the MVP. The audit trail and budget enforcement are still present, just not engine-enforced. v1.2 adds engine self-guarding. |
| DEV-2 | ai-supervisory-interface.md Section 7.5 | Supervisor events use backend ingest (HTTP) for hash chain, not local JSONL. | The supervisor runs on hosted infrastructure (Mode 2/3). Writing local JSONL on the server and then ingesting it is redundant. The backend ingest handler computes and verifies the hash chain. |
| DEV-3 | ai-supervisory-interface.md Section 6.2 (`deny_spike`) | Minimum deny count raised from unspecified to 5 (spec just says "deny rate > 3x baseline"). | Avoids noisy triggers on small runs (e.g., 2 denies out of 3 calls = 66% deny rate, which would trigger on a 10% baseline). |
