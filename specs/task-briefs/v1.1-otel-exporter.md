## 1. Architectural Gates Resolved

This section resolves every open gate for the OTel Exporter before
implementation begins.

### Gate OTEL-G1: OTel SDK Dependencies

**Decision:** Use the `@opentelemetry/sdk-trace-base` package (not
`sdk-trace-node`), combined with the OTLP exporters.

**Rationale:** `sdk-trace-base` is the platform-agnostic trace SDK that
works in both Node.js and Bun. `sdk-trace-node` adds Node.js-specific
auto-instrumentation (context managers, `AsyncLocalStorage`), which we do
not need -- the exporter is a standalone process, not a library integrated
into someone else's app. Using `sdk-trace-base` avoids the Node.js-specific
dependency chain and works correctly under Bun.

**Exact dependencies:**

```json
{
  "dependencies": {
    "@opentelemetry/api": "^1.9.0",
    "@opentelemetry/sdk-trace-base": "^1.30.0",
    "@opentelemetry/exporter-trace-otlp-http": "^0.57.0",
    "@opentelemetry/exporter-trace-otlp-grpc": "^0.57.0",
    "@opentelemetry/resources": "^1.30.0",
    "@opentelemetry/semantic-conventions": "^1.30.0"
  }
}
```

**Why both gRPC and HTTP exporters:** The spec (Section 8.3) requires both.
The operator selects the transport via `OTEL_EXPORTER_OTLP_PROTOCOL` (default
`http/protobuf`). The implementation lazy-loads the selected exporter so that
the unused protocol's dependency is never initialized at runtime.

**Default changed from spec recommendation:** The spec Section 9.2 suggests
gRPC as the default protocol. This brief changes the default to `http/protobuf`
because:
1. `http/protobuf` works in more environments (proxies, firewalls, serverless).
2. The OTel community is converging on HTTP as the primary transport.
3. gRPC requires a native addon (`@grpc/grpc-js`) which adds install
   complexity and may fail under Bun.
4. Operators who want gRPC explicitly set `OTEL_EXPORTER_OTLP_PROTOCOL=grpc`.

**Console exporter:** Include `@opentelemetry/sdk-trace-base`'s built-in
`ConsoleSpanExporter` as a debug mode (no additional dependency). Activated
via `--console` flag or `OTEL_EXPORTER=console` environment variable.

### Gate OTEL-G2: Span Structure

**Decision:** One span per event. Flat hierarchy under a root span.

This was already resolved in the spec (Sections 2.2-2.3) but is restated
here for clarity:

- Every LoopStorm event (JSONL line) becomes exactly one OTel span.
- `run_started` becomes the root span (parent_span_id = none).
  Its span_id is derived from seq=0 (synthetic).
- All other events become child spans of the root span. Their span_ids
  are derived from their actual `seq` values.
- The hierarchy is flat -- no nesting beyond root -> children.
- Budget warnings and loop detections are ALSO attached as OTel Span Events
  on the root span, but only as a convenience (they already exist as
  standalone child spans).

**Buffering strategy:** Buffer all spans per trace (keyed by `run_id`).
Flush the entire trace when:
1. A `run_ended` event is received (set root span end time and status), OR
2. A configurable timeout elapses without `run_ended` (default: 5 minutes),
   in which case the root span is emitted with zero duration and UNSET status.

This is spec Section 11.3 option (b) for completed runs, option (c) for
incomplete runs.

### Gate OTEL-G4: JSONL Reader Design

**Decision:** Streaming line reader with tail-follow via `fs.watchFile`,
not `chokidar` or `fs.watch`.

**Rationale:**
- `fs.watchFile` uses polling (stat-based). It is slower than inotify/kqueue
  but works reliably on all platforms including Windows NTFS, NFS mounts,
  and Docker bind-mounts.
- `fs.watch` (inotify/kqueue) is unreliable on NFS, Docker volumes, and
  some Windows configurations. The spec explicitly calls this out (Section
  6.9 risk area 2).
- `chokidar` adds a dependency (and its own platform compatibility layer).
  The polling behavior of `fs.watchFile` is sufficient because the exporter
  is not latency-sensitive (seconds of delay are acceptable).
- `fs.watchFile` polling interval: 1000ms (configurable via
  `poll_interval_ms`).

**Reader architecture:**

```
[JSONL File] --poll--> [JsonlReader] --events--> [TraceBuffer] --flush--> [SpanExporter]
```

1. `JsonlReader` opens the file, reads to the current end, parses lines.
2. It stores its cursor (byte offset) in memory.
3. On each poll, it stats the file. If `size > cursor`, it seeks to `cursor`,
   reads new bytes, splits on `\n`, and parses complete lines.
4. Partial lines (file written mid-line) are buffered until the next poll.
5. For crash recovery: the cursor is persisted to a `.cursor` file alongside
   the JSONL file (e.g., `audit.jsonl.otel-cursor`). On startup, if the
   cursor file exists and the JSONL file size is >= the cursor value, resume
   from the cursor. Otherwise, start from byte 0.
6. For file rotation: if the file size is LESS than the cursor, the file
   was truncated or rotated. Reset cursor to 0 and re-process.

**One-shot batch mode:** In addition to tail-follow, support a `--once` flag
that reads the entire file, exports all spans, and exits. This is useful for
backfilling historical data and for CI integration tests.

### Gate OTEL-G5: Export Targets

**Decision:** Three export targets, selected by configuration.

| Target | Activation | Dependency |
|---|---|---|
| OTLP/HTTP (protobuf) | Default. `OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf` | `@opentelemetry/exporter-trace-otlp-http` |
| OTLP/gRPC | `OTEL_EXPORTER_OTLP_PROTOCOL=grpc` | `@opentelemetry/exporter-trace-otlp-grpc` |
| Console (stdout) | `--console` flag or `OTEL_EXPORTER=console` | Built-in `ConsoleSpanExporter` |

**OTLP endpoint:** Read from `OTEL_EXPORTER_OTLP_ENDPOINT` (standard OTel
env var). Default: `http://localhost:4318` (HTTP) or `http://localhost:4317`
(gRPC).

**OTLP headers:** Read from `OTEL_EXPORTER_OTLP_HEADERS` (standard OTel
env var). Used for API key authentication to SaaS backends (Datadog,
Grafana Cloud, Honeycomb).

**Batch export:** Use OTel SDK's `BatchSpanProcessor` with configurable
batch size (default 100) and flush interval (default 5000ms).

### Gate OTEL-G6: CLI Interface

**Decision:** The exporter is invoked as a standalone CLI process.

**Command:**

```bash
# Tail-follow mode (default)
loopstorm-otel-export --input /path/to/audit.jsonl

# One-shot mode (process and exit)
loopstorm-otel-export --input /path/to/audit.jsonl --once

# With explicit endpoint
loopstorm-otel-export --input audit.jsonl --endpoint http://localhost:4318

# Console output for debugging
loopstorm-otel-export --input audit.jsonl --console

# With gRPC
OTEL_EXPORTER_OTLP_PROTOCOL=grpc loopstorm-otel-export --input audit.jsonl
```

**CLI flags:**

| Flag | Env var | Default | Description |
|---|---|---|---|

### Gate OTEL-G7: Error Handling

**Decision:** The exporter is non-critical. Errors are logged, not fatal.

| Error case | Behavior |
|---|---|
| JSONL file does not exist | Wait and retry every `poll_interval_ms`. Log warning on first attempt. |
| Malformed JSON line | Log warning with line number and byte offset. Skip the line. Continue processing. |
| Missing required fields (`run_id`, `seq`, `event_type`, `hash`, `ts`) | Treat as malformed. Log and skip. |
| Unexpected `schema_version` | Log warning. Process anyway (the attribute mapping may be incomplete but spans are still useful). |
| OTLP connection failure | OTel SDK handles retry with exponential backoff. The `BatchSpanProcessor` queues spans in memory (capped at 2048 default). If the queue fills, oldest spans are dropped. Log warning. |
| File truncated / rotated | Reset cursor to 0. Log info. Re-process from start. |
| Cursor file corrupted | Delete cursor file. Start from byte 0. Log warning. |
| `span_id` all-zero edge case | Set least significant bit to 1 (per spec Section 3.2). |

**No fail-closed requirement.** The exporter is not in the enforcement path.
It is not in the observation plane. It is a telemetry consumer with read-only
access. Its failure mode is "stop exporting" -- it does not affect enforcement,
does not affect the supervisor, and does not affect the JSONL audit log.

### Gate OTEL-G10: Supervisor Span Handling

**Decision:** Supervisor events produce separate traces with span links.
This was already resolved in spec Section 6 but requires implementation
clarity.

**Detection:** Events with `event_type` starting with `supervisor_` belong
to supervisor traces. The exporter detects this by checking `event_type`.

**Separate trace:** Supervisor events use the supervisor's own `run_id` for
`trace_id` derivation -- which is their `run_id` field, same as any other
trace. The fact that it is a supervisor trace is encoded in:
- Resource attribute `service.name` = `"loopstorm-supervisor"` (when
  `agent_role` = `"supervisor"`)
- The span link to the triggering agent's root span

**Span link construction:** When processing a `supervisor_run_started`
event that contains `trigger_run_id`:
1. Derive the trigger trace_id: `traceId(trigger_run_id)`
2. Derive the trigger root span_id: `spanId(trigger_run_id, 0)`
3. Attach a Link to the supervisor's root span with:
   - `trace_id` = trigger trace_id
   - `span_id` = trigger root span_id
   - `attributes` = `{ "loopstorm.link_type": "supervisor_trigger" }`

**Resource detection:** If the first event in a trace has `agent_role` =
`"supervisor"` (or `event_type` = `supervisor_run_started`), set
`service.name` to `"loopstorm-supervisor"`. Otherwise, use the configured
service name (default `"loopstorm-engine"`).

### Task OTEL-2: ID Derivation Functions

**Location:** `apps/otel-exporter/src/ids.ts`

**Action:** Implement `traceId()` and `spanId()` per spec Section 3.1-3.2.

```typescript
// SPDX-License-Identifier: MIT

import { createHash } from "node:crypto";

/**
 * Derive an OTel trace_id from a LoopStorm run_id (UUID).
 * Direct byte mapping: strip hyphens, lowercase.
 * Spec: otel-span-mapping.md Section 3.1
 */
export function traceId(runId: string): string {
  return runId.replace(/-/g, "").toLowerCase();
}

/**
 * Derive an OTel span_id from a (run_id, seq) pair.
 * SHA-256(run_id_bytes || seq_be_bytes), take first 8 bytes.
 * Spec: otel-span-mapping.md Section 3.2
 *
 * For the root span (run_started), use seq=0 (synthetic).
 * For all other events, use the event's actual seq value.
 */
export function spanId(runId: string, seq: number): string {
  const runIdHex = runId.replace(/-/g, "").toLowerCase();
  const runIdBytes = Buffer.from(runIdHex, "hex");
  const seqBytes = Buffer.alloc(8);
  seqBytes.writeBigUInt64BE(BigInt(seq));
  const payload = Buffer.concat([runIdBytes, seqBytes]);
  const hash = createHash("sha256").update(payload).digest();
  let result = hash.subarray(0, 8);

  // Non-zero guarantee (spec Section 3.2)
  if (result.every((b) => b === 0)) {
    result[7] = 1;
  }

  return result.toString("hex");
}
```

### Task OTEL-6: Span Builder (Core)

**Location:** `apps/otel-exporter/src/span-builder.ts`

**Action:** Implement the core event-to-span mapping logic.

This is the largest module. It receives parsed JSONL events and produces
OTel `ReadableSpan` objects.

**Key behaviors:**

1. **Root span creation:** When a `run_started` event arrives, create a root
   span with `span_id = spanId(run_id, 0)` (synthetic seq=0). Start time =
   `ts`. End time = unknown (set later). Status = UNSET.

2. **Child span creation:** For every other event, create a child span with
   `parent_span_id = spanId(run_id, 0)` (the root span). `span_id =
   spanId(run_id, event.seq)`. Start time = `ts`. End time determined by
   `latency_ms` (for `policy_decision`) or same as start time (zero duration).

3. **Root span finalization:** When a `run_ended` event arrives, update the
   root span:
   - `end_time` = `run_ended.ts`
   - `status` = based on `run_status` (Section 5.2 in spec)

4. **OTel Span Events on root:** When processing `budget_soft_cap_warning`,
   `budget_exceeded`, or `loop_detected`, also attach an OTel Span Event to
   the root span (per spec Appendix C):
   - `budget_soft_cap_warning` -> event name `loopstorm.budget_warning`
   - `budget_exceeded` -> event name `loopstorm.budget_exceeded`
   - `loop_detected` -> event name `loopstorm.loop_detected`

5. **Supervisor span links:** When processing `supervisor_run_started` with
   `trigger_run_id`, attach a Link to the root span of the supervisor trace
   (see Gate OTEL-G10).

6. **Span name:** `loopstorm.{event_type}` (e.g., `loopstorm.policy_decision`).

**Span construction approach:** Use the OTel SDK's programmatic span
building. Create spans from the `@opentelemetry/sdk-trace-base` `Tracer`
using `tracer.startSpan()` with explicit timestamps, then set attributes
and end the span.

**Alternative (recommended for testability):** Construct `ReadableSpan`
objects directly rather than using the Tracer API. This avoids needing an
active trace context and makes unit testing simpler. The `BasicTracerProvider`
with `SimpleSpanProcessor` + `InMemorySpanExporter` pattern works well for
this.

**Acceptance criteria:**
- AC-6-1: `run_started` creates a root span with span_id from seq=0.
- AC-6-2: `policy_decision` creates a child span with `latency_ms` as duration.
- AC-6-3: `run_ended` finalizes root span with correct end_time and status.
- AC-6-4: Budget/loop events attached as OTel Span Events on root span.
- AC-6-5: `supervisor_run_started` with `trigger_run_id` produces a span link.
- AC-6-6: All 12 event types produce correctly named and attributed spans.
- AC-6-7: BT fields appear as `loopstorm.telemetry.*` attributes on policy_decision spans.

### Task OTEL-7: Trace Buffer

**Location:** `apps/otel-exporter/src/trace-buffer.ts`

**Action:** Implement per-trace span buffering with flush-on-complete and
timeout-based flush.

**Buffer design:**

```typescript
class TraceBuffer {
  // Map from run_id to buffered trace state
  private traces: Map<string, BufferedTrace>;

  // Add an event to the buffer. Returns spans to flush (empty array
  // if trace is not yet complete).
  addEvent(event: ParsedEvent): ReadableSpan[];

  // Check for timed-out traces. Returns spans to flush.
  flushTimedOut(now: number, timeoutMs: number): ReadableSpan[];

  // Flush all buffered traces (for shutdown).
  flushAll(): ReadableSpan[];
}

interface BufferedTrace {
  runId: string;
  rootSpan: ReadableSpan | null;  // Created on run_started
  childSpans: ReadableSpan[];
  rootSpanEvents: TimedEvent[];   // Budget warnings, loop detections
  firstEventTime: number;         // For timeout calculation
  resource: Resource;
  completed: boolean;             // Set on run_ended
}
```

**Flush triggers:**
1. `run_ended` event received: finalize root span, flush all spans for
   this trace.
2. Buffer timeout elapsed (default 5 minutes from first event): flush all
   spans with root span at zero duration and UNSET status.
3. Shutdown signal: flush all buffered traces immediately.

**Memory bound:** Maximum 1000 concurrent traces in the buffer. If exceeded,
flush the oldest trace (by first event time). This prevents unbounded memory
growth if many runs never complete.

**Acceptance criteria:**
- AC-7-1: Complete trace (run_started + events + run_ended) flushed as single batch.
- AC-7-2: Incomplete trace flushed after timeout.
- AC-7-3: Multiple concurrent traces buffered independently.
- AC-7-4: Buffer cap (1000) enforced; oldest trace evicted when exceeded.
- AC-7-5: `flushAll()` returns all buffered spans.

### Task OTEL-8: JSONL Reader

**Location:** `apps/otel-exporter/src/jsonl-reader.ts`

**Action:** Implement a tail-follow JSONL file reader with cursor persistence.

**Key behaviors:**

1. **Initial read:** Open file, read all existing lines from byte 0
   (or from cursor if resuming), parse each line as JSON, emit events.

2. **Tail-follow:** After initial read, poll the file every `poll_interval_ms`
   using `fs.stat()`. If file size > cursor, seek to cursor, read new bytes.

3. **Partial line handling:** If the last chunk read does not end with `\n`,
   buffer it and prepend to the next chunk.

4. **Cursor persistence:** After processing each batch of lines, write the
   current byte offset to `{input_path}.otel-cursor`. On restart, resume
   from this offset.

5. **File rotation:** If file size < cursor, the file was truncated or
   rotated. Log an info message, reset cursor to 0, re-process from start.

6. **Malformed lines:** Parse each line with `JSON.parse()` inside a
   try-catch. On parse failure, log a warning with the line number and
   skip. On missing required fields, log and skip.

**Event emitter pattern:** The reader emits parsed events to a callback
function provided by the caller:

```typescript
type EventHandler = (event: ParsedEvent) => void;

class JsonlReader {
  constructor(
    inputPath: string,
    onEvent: EventHandler,
    options?: { pollIntervalMs?: number }
  );

  // Start reading (initial + tail-follow). Returns when stop() is called.
  start(): Promise<void>;

  // Process file once and return. For --once mode.
  processOnce(): Promise<void>;

  // Stop the reader.
  stop(): void;
}
```

**Acceptance criteria:**
- AC-8-1: Reads all existing lines on startup.
- AC-8-2: Detects new lines appended to the file.
- AC-8-3: Handles partial lines (no premature parse).
- AC-8-4: Persists cursor to `.otel-cursor` file.
- AC-8-5: Resumes from cursor on restart.
- AC-8-6: Detects file rotation (size < cursor) and resets.
- AC-8-7: Skips malformed lines with a warning log.
- AC-8-8: `processOnce()` reads file and returns (for --once mode).

## 11. Mode 0 Compatibility

The OTel exporter is fully Mode 0 compatible with a local OTel collector:

```
[Engine] -> [JSONL file]
                |
          [loopstorm-otel-export --input audit.jsonl]
                |
          [Local Jaeger / Grafana Tempo]
```

No network access to external services required. The exporter and collector
both run on the same machine.

**Note:** Mode 0 is not the primary use case for OTel export (spec Section
10.1 acknowledges this). Mode 0 users typically use the CLI for analysis.
OTel export in Mode 0 is for operators who already run a local observability
stack.

## 13. Architectural Decisions Confirmed

| Decision | Rationale | Source |
|---|---|---|
| JSONL reader only in v1.1 | Mode 0 priority; event store reader deferred to v1.2 | ADR-014 Gate 6 |
| TypeScript implementation | Aligns with TS stack; OTel JS SDK is mature | v1.1 plan Section 6 |
| `sdk-trace-base` not `sdk-trace-node` | Platform-agnostic; Bun compatible | This brief, Gate OTEL-G1 |
| HTTP/protobuf default (not gRPC) | More portable; gRPC has Bun issues | This brief, Gate OTEL-G1 |
| One span per event, flat hierarchy | Spec Section 2.2-2.3; simplest correct mapping | Spec |
| Buffer per trace, flush on complete | Spec Section 11.3; correct root span duration | Spec |
| `fs.watchFile` for tail-follow | Cross-platform reliability over performance | This brief, Gate OTEL-G4 |
| No LoopStorm package dependencies | Maximally isolated; reads files only | This brief |
| Console exporter for debug | No external dependency; useful for development | This brief, Gate OTEL-G5 |
