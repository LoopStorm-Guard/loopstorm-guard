## 3. Field Computation Reference (Already Implemented)

These algorithms are already implemented in `apps/engine/src/telemetry.rs`.
This section is included for implementer reference when writing backend and
integration tests.

### 3.1 `call_seq_fingerprint`

- Maintain a `VecDeque<(tool, args_hash)>` with max capacity 5.
- On each call: push `(tool, args_hash)`, evict oldest if len > 5.
- Build payload: join entries as `"{tool}:{args_hash}"` with `\n` delimiter.
- Return `hex_lower(SHA-256(UTF-8(payload)))`.
- First call: window size 1 (still produces a valid fingerprint).

### 3.2 `inter_call_ms`

- Store `Instant::now()` at request receipt per run.
- First call: return `0`.
- Subsequent: `(now - prev).as_millis() as u64`.
- Uses monotonic clock, NOT wall clock.

### 3.3 `token_rate_delta`

- Only computed when `input_tokens` or `output_tokens` is present.
- Omitted entirely when no token data is provided (not `null`, not `0`).
- First call with token data: returns `1.0`.
- Subsequent: `call_tokens / rolling_average`, rounded to 6 decimal places.
- `rolling_average = total_token_sum / token_call_count` (before current call).
- Division by zero guard: if `rolling_average == 0`, return `call_tokens` as raw float.

### 3.4 `param_shape_hash`

- Stateless (no per-run state needed).
- If `args_redacted` is an object: extract top-level keys, sort lexicographically,
  join with `\n`, SHA-256.
- If null: hash `"null"`.
- If array: hash `"array"`.
- If string/number/boolean: hash the type tag string.
- Empty object `{}`: hash empty string (SHA-256 of zero bytes).

## 8. Backward Compatibility

### 8.1 v1.0 Shims with v1.1 Engine

**Works.** v1.0 shims send `DecisionRequest` messages that lack
`input_tokens`/`output_tokens` (or include them). The engine computes BT
fields regardless. The IPC response is unchanged. The shim is unaware BT
fields exist.

### 8.2 v1.1 Engine JSONL with v1.0 CLI Verifier

**BREAKS.** This is the known backward-compat issue (spec Section 9.1, risk
R3 in the implementation plan). The chain verification algorithm hashes ALL
fields (except `hash` and `hash_prev`). A v1.0 verifier that does not know
about BT fields will recompute the event hash differently because the
engine's `AuditEvent` serialization includes BT fields in the payload.

Wait -- actually, re-reading the CLI `verify.rs` implementation: it uses
**raw-string stripping** of the `hash` and `hash_prev` fields from the
actual JSONL line bytes. It does NOT deserialize, strip unknown fields, then
re-serialize. This means:

1. The verifier reads the raw JSONL line.
2. It strips the exact `,"hash":"<64-hex>"` and `,"hash_prev":"<64-hex>"`
   substrings from the raw line.
3. It SHA-256 hashes the remaining raw bytes.
4. It compares against the stored `hash`.

Because the engine also computes the hash by serializing the `AuditEvent`
with `hash=None` and `hash_prev=None` (which `skip_serializing_if` omits),
the raw-string approach produces the same result regardless of which fields
are present -- as long as the verifier strips the same fields the engine
omitted.

**Conclusion: v1.0 CLI verify DOES work with v1.1 JSONL files**, because the
verifier operates on raw bytes and BT fields are simply part of the payload
that gets hashed. The verifier does not need to know about BT fields at all.

**This contradicts the risk stated in spec Section 9.1 and implementation
plan risk R3.** The risk analysis assumed a Value round-trip verification
approach. The actual implementation uses raw-string stripping, which is
field-agnostic. This is a positive finding.

**However:** Task BT-G6 must still verify this with an explicit test.

### 8.3 v1.0 Backend Ingest with v1.1 JSONL

**Partially works.** The Zod schema uses `z.object({...})` without
`.strict()`, so unknown fields are silently stripped (Zod default behavior).
The BT fields in the JSONL line will be parsed but discarded. The `raw_line`
column stores the original JSONL line including BT fields, so no data is
permanently lost. Chain verification uses `raw_line`, not the structured
columns.

After applying tasks G1-G3, the backend will correctly parse and store BT
fields.

### 8.4 v1.1 Backend with v1.0 JSONL

**Works.** All 4 new DB columns are nullable. v1.0 events store NULL.
