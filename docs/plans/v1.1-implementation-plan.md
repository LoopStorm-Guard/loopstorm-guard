## 1. Executive Summary

v1.1 adds four features that have complete specification documents but zero
implementation code. The features span the full stack: Rust engine changes,
a new TypeScript package, backend API extensions, database migrations,
frontend UI, and CI pipeline additions.

|---|---|---|---|---|
| Behavioral Telemetry | `specs/behavioral-telemetry.md` | **M** | Rust | 4 new fields on `policy_decision` events |

## 2. Dependency Graph

```
                  Behavioral Telemetry (BT)
                      |
                      v
         +---> OTel Span Mapping (OTEL) ---+
         |            ^                      |
         |            |                      |
         |    AI Supervisor (SUP) ---------->+
         |            ^
         |            |
         +--- MCP Proxy Mode (MCP)
```

**Dependency details:**

| From | To | Nature of dependency |
|---|---|---|
| BT | OTEL | OTel maps BT fields to `loopstorm.telemetry.*` attributes. OTEL can be built in parallel but needs BT's fields defined in the event schema. |
| BT | SUP | Supervisor's `read_run_events` returns BT fields. Supervisor's `compute_risk_score` and `query_similar_runs` consume `call_seq_fingerprint`. |
| MCP | OTEL | MCP-proxied events appear as standard `policy_decision` events. OTEL exports them with no special handling. No code dependency, but integration testing requires both. |
| MCP | SUP | Supervisor can analyze MCP-proxied runs. No code dependency. |
| SUP | OTEL | OTel Section 6 defines supervisor span mapping. OTEL must handle `supervisor_*` event types. |

**The critical dependency is:** Behavioral Telemetry must ship first (or
concurrently) because it changes the engine's per-call output that three
other features consume.

## 3. Recommended Implementation Order

### Phase 1: Behavioral Telemetry (BT)

**Why first:** Changes the engine (Rust), which is the foundation. The
four new fields appear in the JSONL audit log and must exist before the
OTel exporter can map them or the supervisor can read them. Engine changes
have the longest CI feedback loop (cross-compilation).

**Duration:** 1-1.5 weeks

### Phase 2: MCP Proxy Mode (MCP)

**Why second:** Standalone TypeScript package with no dependencies on
BT's Rust changes at the code level (the proxy sends `DecisionRequest`
messages; the engine computes BT fields internally). Can be started during
Phase 1's review period.

**Duration:** 1.5-2 weeks

### Phase 3: OTel Span Mapping (OTEL)

**Why third:** The exporter is a read-only consumer. It needs BT fields
to exist in the event schema (Phase 1) and benefits from MCP-proxied
events existing for integration tests (Phase 2). It has no consumers --
nothing depends on it.

**Duration:** 1-1.5 weeks

### Phase 4: AI Supervisor (SUP)

**Why last:** The most complex feature. Depends on BT fields being present
in events it reads. Requires backend API extensions, a new LLM integration
layer, the supervisor policy pack, trigger evaluation logic, and frontend
UI for the approval workflow. Benefits from all three preceding features
being stable.

**Duration:** 2-3 weeks

### 4.5 Implementation Steps

#### Step BT-1: Create `telemetry.rs` module

Create `apps/engine/src/telemetry.rs` with:

```
pub struct TelemetryState {
    /// Ring buffer of (tool, args_hash) tuples, capacity N=5
    call_window: VecDeque<(String, String)>,
    /// Monotonic timestamp of the last DecisionRequest receipt
    prev_request_instant: Option<Instant>,
    /// Cumulative token sum across all calls in this run
    total_token_sum: u64,
    /// Count of calls that provided token data
    token_call_count: u64,
}

pub struct TelemetryFields {
    pub call_seq_fingerprint: String,
    pub inter_call_ms: u64,
    pub token_rate_delta: Option<f64>,
    pub param_shape_hash: String,
}

impl TelemetryState {
    pub fn new() -> Self { ... }
    pub fn compute(&mut self, tool: &str, args_hash: &str,
                   args_redacted: Option<&serde_json::Value>,
                   input_tokens: Option<u64>,
                   output_tokens: Option<u64>) -> TelemetryFields { ... }
}
```

Key implementation details:
- `call_seq_fingerprint`: `VecDeque<(String, String)>` with max capacity 5.
  Push `(tool, args_hash)`, evict oldest if len > 5. Join with `\n` as
  `"{tool}:{args_hash}\n{tool}:{args_hash}"`, SHA-256 the UTF-8 bytes.
- `inter_call_ms`: Store `Instant::now()` per call. First call returns 0.
  Subsequent calls return `(now - prev).as_millis()` as u64.
- `token_rate_delta`: Only computed when `input_tokens` or `output_tokens`
  is `Some`. Use `f64` division, round to 6 decimal places.
- `param_shape_hash`: Extract keys from `args_redacted` if it is a JSON
  object, sort lexicographically, join with `\n`, SHA-256. For non-object
  types, hash the type tag string.

#### Step BT-3: Integrate telemetry into `enforce()`

In `apps/engine/src/lib.rs`:

1. Add `telemetry_states: HashMap<String, TelemetryState>` to
   `EnforcementContext`.
2. In `enforce()`, before building the `AuditEvent`:
   - Look up (or create) `TelemetryState` for `request.run_id`.

not a v1.1 requirement -- a `HashMap<String, TelemetryState>` on
`EnforcementContext` is sufficient.

#### Step BT-4: Update event schema

In `schemas/events/event.schema.json`, add the 4 properties as defined

schema changed (it should NOT -- only event schema changes), and run
`scripts/sync-schemas.sh`.

**IMPORTANT:** The event schema has `"additionalProperties": false`. The
4 new fields MUST be added as explicit properties, not as
`additionalProperties: true`.

#### Step BT-5: Database migration

Create a new Drizzle migration adding 4 nullable columns to `events`:

```sql

ALTER TABLE events ADD COLUMN inter_call_ms integer;
ALTER TABLE events ADD COLUMN token_rate_delta double precision;
ALTER TABLE events ADD COLUMN param_shape_hash text;
```

Update `packages/backend/src/db/schema.ts` to include these columns.

#### Step BT-6: Test vectors

1. Create `tests/fixtures/behavioral-telemetry-vectors.json` with all
   vectors from spec Section 8 (CSF-1 through CSF-4, ICM-1 through
   ICM-4, TRD-1 through TRD-6, PSH-1 through PSH-5).
2. Write Rust tests that load and verify against these vectors.
3. Write at least one integration test that verifies BT fields appear
   in the JSONL output of a multi-call run.

#### Step BT-7: CLI verifier compatibility

Verify that `apps/cli/` `verify` and `replay` commands handle JSONL
files containing BT fields. The chain verification algorithm hashes ALL
fields (except `hash` and `hash_prev`). A v1.0 verifier will compute a
different hash for v1.1 events. The CLI MUST be updated to v1.1
simultaneously with the engine.

**This is the one backward-compat risk.** See spec Section 9.1. The
verifier must be rebuilt from the same commit that adds BT fields.

### 4.7 Complexity: **M** (Medium)

The algorithms are straightforward (SHA-256, arithmetic, ring buffer).
The complexity is in the number of touch points: engine, schema, DB

### 4.8 Architectural Decisions / Gates

| Gate | Decision | Status |
|---|---|---|
| Schema version bump? | No. Additive optional fields within schema_version 1. | **Resolved** (spec Section 4) |
| Where computed? | Engine only. Shims do not compute BT fields. | **Resolved** (spec Section 7) |
| Window size configurable? | No. N=5 is a compile-time constant. | **Resolved** (spec Appendix A, OQ-BT-3) |

### 4.9 Risk Areas

1. **Chain verification backward compat:** v1.0 CLI cannot verify v1.1
   JSONL files. Mitigation: document in release notes; CLI must be
   updated to v1.1 alongside the engine.
2. **`token_rate_delta` float precision:** IEEE 754 rounding. The spec
   mandates 6 decimal places via `(delta * 1_000_000.0).round() / 1_000_000.0`.
   Test with the exact spec vectors.
3. **`param_shape_hash` key ordering:** Must be UTF-16 code unit order
   (same as JCS). For ASCII keys (common case), this is byte order. Add
   a test with non-ASCII keys to verify.

`apps/mcp-proxy/src/config.ts`:

#### Step MCP-4: Implement engine client

Either import from `packages/ipc-client` (after Step MCP-2) or duplicate.

The proxy needs a persistent connection with reconnect logic (exponential
backoff: 100ms, 200ms, 400ms, ..., capped at 5s). During reconnection,
all `tools/call` requests are denied (fail-closed). The shim's
`EngineConnection` already supports this pattern but may need the
reconnect-with-backoff addition.

#### Step MCP-5: Implement MCP-to-LoopStorm mapping

`apps/mcp-proxy/src/mapping.ts`:
- `mcpToolCallToDecisionRequest()`: Implements the mapping table from
  spec Section 3.1.
- `decisionResponseToMcpResponse()`: Implements spec Section 3.2.
- MCP error code constants from spec Section 3.3.

Key mapping rules:
- `params.name` -> `tool` (direct copy, or `prefix.name` if prefixing enabled)
- `params.arguments` -> `args_hash` (via JCS + SHA-256)
- Proxy-generated `run_id` (UUID v7 per connection)
- Proxy-tracked `seq` (per `run_id`, starts at 1)
- `ts` = ISO 8601 at interception time
- `schema_version` = 1

#### Step MCP-6: Implement core proxy

`apps/mcp-proxy/src/proxy.ts`:

This is the main orchestrator. Uses `@modelcontextprotocol/sdk` to:
1. Create an MCP server (agent-facing, stdio or HTTP transport).
2. Create MCP clients for each upstream server.
3. Handle `tools/list`: merge upstream tool lists, apply prefixing.
4. Handle `tools/call`: translate to `DecisionRequest`, consult engine,
   forward if allowed, return error if denied.
5. Pass through all other MCP features (resources, prompts, sampling,
   notifications) per spec Section 7.
6. Handle `run_id` assignment per spec Section 6.3.

The proxy generates a UUID v7 per MCP client connection. Override via
`params._meta.loopstorm_run_id` on first `tools/call`.

#### Step MCP-7: Implement CLI entry point

`apps/mcp-proxy/bin/loopstorm-mcp-proxy.ts`:

- `--transport stdio|http` (override config)
- `--port <number>` (override config)
- `--version`, `--help`
- Add `"bin"` field to `package.json`.

The MCP SDK integration is the complexity driver. MCP supports multiple

### 5.8 Architectural Decisions / Gates

| Gate | Decision | Status |
|---|---|---|
| Implementation language | TypeScript | **Resolved** (OQ-MCP-1) |
| `run_id` generation | UUID v7 per connection, override via `_meta` | **Resolved** (OQ-MCP-2) |
| `seq` tracking | Proxy tracks per `run_id` | **Resolved** (OQ-MCP-3) |
| MCP spec version | 2025-03-26 | **Resolved** (OQ-MCP-4) |
| Fail-closed (no `fail_open`) | Unconditionally fail-closed | **Resolved** (spec Section 3.4) |
| **NEW GATE: Shared IPC client package** | Extract to `packages/ipc-client/`? | **Needs resolution** |

### 5.9 Risk Areas

1. **MCP SDK stability:** `@modelcontextprotocol/sdk` is < 2.0. API may
   change. Pin to `>=1.10.0 <2.0.0`.
2. **Upstream server process management:** stdio transport requires the
   proxy to spawn and manage upstream MCP server subprocesses. Process
   lifecycle (crash recovery, graceful shutdown) is complex.
3. **MCP connection-to-run_id mapping:** The spec ties `run_id` to the
   MCP connection. For stdio, a "connection" is the process lifetime.
   For HTTP, it is less clear (sessions? individual requests?). The
   MCP SDK's session management determines this.
4. **UUID v7 generation in TypeScript:** No native `uuid7()`. Use the
   `uuidv7` npm package or implement from `crypto.getRandomValues()` +
   timestamp.

#### Step OTEL-2: Implement ID derivation

`apps/otel-exporter/src/ids.ts`:

```typescript
function traceId(runId: string): string {
  // Strip hyphens from UUID, lowercase
  return runId.replace(/-/g, "").toLowerCase();
}

function spanId(runId: string, seq: number): string {
  // SHA-256(run_id_bytes || seq_be_bytes), take first 8 bytes
  const runIdBytes = Buffer.from(runId.replace(/-/g, ""), "hex");
  const seqBytes = Buffer.alloc(8);
  seqBytes.writeBigUInt64BE(BigInt(seq));
  const payload = Buffer.concat([runIdBytes, seqBytes]);
  const hash = createHash("sha256").update(payload).digest();
  return hash.subarray(0, 8).toString("hex");
}
```

Test against spec Section 3.2 vectors.

#### Step OTEL-3: Implement span builder

`apps/otel-exporter/src/span-builder.ts`:

Core mapping logic per spec Section 4:
- Identity fields -> trace_id, span_id
- `event_type` -> span name (prefixed with `loopstorm.`)
- `decision` -> SpanStatus (Section 5)
- All event fields -> span attributes (Section 4.1-4.9)
- `latency_ms` -> span duration
- Root span duration from `run_ended.ts - run_started.ts`

**Buffering strategy (spec Section 11.3):** Buffer spans per trace.
Flush when `run_ended` is received or after a configurable timeout
(default: 5 minutes).

2. **JSONL tail-follow on Windows:** Node.js `fs.watch` behavior differs

3. **Root span duration for incomplete runs:** Runs that never emit
   `run_ended` (crashed, abandoned) will have zero-duration root spans.
   This is acceptable per spec Section 11.3.
4. **Large JSONL files:** The reader must handle files that grow
   continuously. Memory-efficient line-by-line reading, not `readFile()`.

#### Sub-Phase SUP-A: Backend Extensions (1 week)

##### Step SUP-A1: Trigger evaluation logic

Create `packages/backend/src/lib/triggers.ts`:

```typescript
interface TriggerCondition {
  type: string; // "terminated_budget" | "denied_decisions" | etc.
  check(run: Run, events: Event[]): boolean;
}

function evaluatePostRunTriggers(run: Run, events: Event[]): string[];
function evaluateMidRunTriggers(run: Run, events: Event[], newEvent: Event): string[];
```

Post-run triggers (evaluated on `run_ended` ingest):
- `terminated_budget`: `run.status === "terminated_budget"`
- `terminated_loop`: `run.status === "terminated_loop"`
- `abandoned`: `run.status === "abandoned"`
- `deny_decisions`: any event with `decision === "deny"`
- `high_cost`: `run.total_cost_usd > 0.80 * budget_hard_cap`

Mid-run triggers (evaluated per event ingest):
- `budget_warning`: current budget > 70% of hard cap
- `loop_detected`: event_type === "loop_detected"
- `deny_spike`: deny rate > 3x agent baseline
- `repeated_rule`: same `rule_id` fires 3+ times

##### Step SUP-A2: Agent baseline API

Add to `packages/backend/src/trpc/routers/supervisor-tools.ts`:

```typescript
supervisorTools.getAgentBaseline: protectedProcedure
  .input(z.object({ agent_name: z.string(), lookback_days: z.number().default(30) }))
  .query(async ({ input, ctx }) => {
```

##### Step SUP-B3: Supervisor session manager

`apps/supervisor/src/session.ts`:

```typescript
class SupervisorSession {
  readonly runId: string;        // Supervisor's own run_id
  readonly triggerRunId: string;  // The agent run that triggered this session
  readonly trigger: string;       // Trigger type

  constructor(triggerRunId: string, trigger: string, config: SupervisorConfig);

  /** Run the supervisor session to completion. */
  async execute(): Promise<void>;
}
```

The session:
1. Generates a UUID v7 for its own `run_id`.
2. Creates a `loopstorm.wrap()` guard with `supervisor-policy.yaml` and
   the supervisor engine socket.
3. Emits `supervisor_run_started` event.
4. Enters the LLM tool-use loop:
   a. Sends system prompt + trigger context to the LLM.
   b. LLM returns tool calls.
   c. For each tool call: guard checks the call, then execute the tool.
   d. Return tool results to the LLM.
   e. Repeat until LLM stops or budget is exhausted.
5. Emits `supervisor_tool_call` events for each tool invocation.
6. Emits `supervisor_proposal_created` / `supervisor_escalation_created`
   as appropriate.
7. Session ends.

### 7.9 Risk Areas

1. **LLM integration complexity:** Claude tool-use API requires careful
   handling of multi-turn conversations, tool results, and error
   recovery. If the LLM produces malformed tool calls, the session must
   gracefully degrade.
2. **Trigger evaluation performance:** Mid-run triggers evaluate on every
   event ingest. The `deny_spike` trigger requires computing the agent's
   baseline deny rate, which is a database query. Caching is essential.
3. **Supervisor budget enforcement:** The supervisor's own engine
   instance must be configured and running. Deployment complexity.
4. **LLM cost management:** Even with haiku-class models, poorly tuned
   prompts can burn budget quickly. The $2.00 cap is a safety net, not
   an optimization target.
5. **Testing difficulty:** Full supervisor integration tests require a
   running LLM API (or a mock). Tests should use a mock LLM that returns
   deterministic tool calls.
6. **`@anthropic-ai/sdk` license boundary:** The SDK is MIT. It goes in
   `apps/supervisor/` (MIT). It must NEVER be imported from
   `packages/backend/` (AGPL). The trigger dispatch code in the backend
   queues a job; the supervisor process (in `apps/supervisor/`) picks it
   up and makes the LLM calls.
7. **Proposal status mapping:** The event schema uses `"pending_approval"`
   but the DB uses `"pending"`. The backend already maps between these.
   The supervisor's event emitter must use `"pending_approval"` in events
   and the backend must write `"pending"` to the DB.

## 12. Cross-Cutting Concerns

### 12.1 SPDX Headers

Every new file must carry the correct SPDX header:
- `apps/*`: `// SPDX-License-Identifier: MIT`
- `packages/ipc-client/*`: `// SPDX-License-Identifier: MIT`
- `packages/backend/*`: `// SPDX-License-Identifier: AGPL-3.0-only`
- `packages/web/*`: `// SPDX-License-Identifier: AGPL-3.0-only`

CI enforces this via `license-check` job.

### 12.2 Biome Formatting

All TypeScript files must pass `biome check`. Use 2-space indentation
(not tabs). JSON schema files must NOT be run through Biome (it would
reformat them and break SHA-256 hashes).

### 12.3 Fail-Closed Invariant

Every new component must respect ADR-002:
- **MCP Proxy:** Fail-closed unconditionally (no `fail_open` option).
- **OTel Exporter:** Not in enforcement path, so not applicable.
- **Supervisor:** Self-guarded via its own engine instance. If the
  supervisor's engine is unavailable, the supervisor cannot run (but this
  does not affect customer enforcement -- it is on the observation plane).

### 12.4 `escalate_to_human` Invariant

The supervisor implementation must verify that:
1. `escalate_to_human` is allowed under the supervisor policy.
2. The supervisor's engine instance has the hardcoded allow.
3. Integration tests verify the invariant end-to-end.

### 12.5 Enforcement/Observation Plane Separation

**The supervisor MUST NOT:**
- Import code from the engine crate
- Connect to the customer engine's UDS socket
- Write to the customer's JSONL audit log
- Modify any enforcement state
- Have any code path that influences enforcement decisions

**Verification:** The `license-boundary` CI job must be extended to check
that `apps/supervisor` does not import from `apps/engine`.

### 12.6 Mode 0 Compatibility

- **Behavioral Telemetry:** Fully Mode 0 compatible. Engine-local.
- **MCP Proxy:** Fully Mode 0 compatible. Local process, UDS, stdio.
- **OTel Exporter:** Mode 0 compatible with a local collector.
- **AI Supervisor:** NOT Mode 0 (requires LLM API access). This is
  documented and expected. Mode 0 provides Stages 1-4; Stage 5 requires
  Mode 2/3.

## Appendix B: Feature Acceptance Criteria Summary

### Behavioral Telemetry (11 ACs)

1. Engine computes `call_seq_fingerprint` per spec algorithm
2. Engine computes `inter_call_ms` using monotonic clock
3. Engine computes `token_rate_delta` with 6 decimal precision
4. Engine computes `param_shape_hash` per spec algorithm
5. All 4 fields appear on `policy_decision` events in JSONL
6. All 4 fields are absent on non-`policy_decision` events
7. `token_rate_delta` is omitted when no token data provided
8. All spec test vectors pass (CSF 1-4, ICM 1-4, TRD 1-6, PSH 1-5)
9. Event schema updated with 4 optional properties
10. DB migration adds 4 nullable columns
11. CLI `verify` handles v1.1 JSONL files correctly

### MCP Proxy (14 ACs)

1. Proxy starts in stdio transport mode
2. Proxy starts in HTTP transport mode (localhost only)
3. `tools/call` intercepted and routed to engine
4. `allow` decision forwards call to upstream
5. `deny` decision returns MCP error -32001
6. `cooldown` decision pauses, then returns MCP error -32002
7. `kill` decision returns MCP error -32003 and closes connection
8. `require_approval` returns MCP error -32004
9. Engine unavailable returns MCP error -32005 (fail-closed)
10. `tools/list` merges upstream tool lists with optional prefixing
11. `run_id` generated per connection (UUID v7)
12. `run_id` override via `_meta.loopstorm_run_id`
13. `args_hash` passes all 12 test vectors
14. Non-`tools/call` MCP features pass through unmodified

### OTel Span Mapping (10 ACs)

1. `trace_id` derived from `run_id` (direct byte mapping)
2. `span_id` derived from `(run_id, seq)` (SHA-256 truncation)
3. All spec `span_id` test vectors pass
4. `run_started` maps to root span
5. `policy_decision` maps to child span with `latency_ms` duration
6. `decision` maps to correct SpanStatus
7. Budget/loop events attached as OTel Span Events on root span
8. Supervisor events map to separate traces with span links
9. BT fields map to `loopstorm.telemetry.*` attributes
10. JSONL reader tail-follows and exports incrementally
