# ADR-004: Client-Generated run_id

**Status:** Adopted
**Date:** 2026-03-13
**Decision authority:** Ricardo (Founder / Principal Architect)

---

## Context

Every guarded run needs a unique identifier (`run_id`) that links all events in that run. The identifier could be generated by:

1. The engine (server-side generation).
2. The shim (client-side generation).
3. The backend (requires network).

Mode 0 (air-gapped, no network) must work without any backend. The engine is a co-process started by the shim. The run_id must be available before the first event is emitted.

---

## Decision

The `run_id` is generated by the **shim (client)** using a UUID v7 (time-ordered, random).

UUID v7 is chosen because:
- It is time-ordered, which means run_ids sort chronologically without additional metadata.
- It contains sufficient randomness (74 bits) to prevent collisions across concurrent agents on the same host.
- It is a standard format supported by every language runtime.

The shim generates the run_id before connecting to the engine and includes it in the first DecisionRequest. The engine adopts it for the run's lifetime. The backend accepts it as-is (idempotent insert on `run_id`).

The shim is responsible for ensuring uniqueness. In practice, UUID v7 collision probability is negligible for any realistic deployment scale.

---

## Consequences

**Positive:**
- Mode 0 works without modification. No network required to generate a run_id.
- The shim knows the run_id before the engine starts, enabling pre-run logging.
- Backend insert is idempotent on run_id — duplicate submissions are safe.
- Time-ordered UUIDs enable efficient database indexing and range queries.

**Negative:**
- A malicious or buggy shim could generate duplicate run_ids. The backend's unique constraint prevents data corruption but would reject the second run's events. This is acceptable for a cooperative system.
- Clock skew on the host affects the time-ordering component. This does not affect correctness, only sort order.

---

## Migration Path

If a future version requires server-assigned run_ids (e.g., for stricter uniqueness guarantees in a multi-tenant hosted environment), the backend can assign a canonical run_id on first event receipt and return it in the ingest response. The client-generated run_id would become a `client_run_id` correlation field. This migration is backward-compatible.
