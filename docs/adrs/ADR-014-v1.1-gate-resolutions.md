# ADR-014: v1.1 Gate Resolutions

**Status:** Adopted
**Date:** 2026-04-02
**Author:** Lead Architect

---

## Context

The v1.1 implementation plan (`docs/plans/v1.1-implementation-plan.md`)
identified seven open architectural gates that must be resolved before
implementation begins. These gates block one or more of the four v1.1
features: Behavioral Telemetry (BT), MCP Proxy (MCP), OTel Exporter
(OTEL), and AI Supervisor (SUP).

The gates are:

1. **Shared IPC client extraction** -- should `packages/ipc-client/` be
   created? Blocks MCP, SUP.
2. **Supervisor system prompt design** -- spec explicitly deferred this.
   Blocks SUP.
3. **Trigger queue mechanism** -- in-process vs. external queue. Blocks
   SUP.
4. **LLM provider abstraction** -- direct SDK vs. wrapper. Blocks SUP.
5. **Risk score algorithm weights** -- how to weight individual signals.
   Blocks SUP.
6. **OTel event store reader scope** -- in v1.1 or deferred? Blocks OTEL.
7. **R7: `escalate_to_human` rule_id mismatch** -- engine uses
   `__builtin_escalate_to_human`, spec uses
   `__builtin_escalate_to_human_allow`. Blocks all.

---

## Decisions

### Gate 1: Shared IPC Client Extraction

**Decision:** YES -- extract `packages/ipc-client/` as a new MIT package.

**What moves:**
- `apps/shim-ts/src/connection.ts` -> `packages/ipc-client/src/connection.ts`
- `apps/shim-ts/src/protocol.ts` -> `packages/ipc-client/src/protocol.ts`
- `apps/shim-ts/src/errors.ts` -> `packages/ipc-client/src/errors.ts`
- `apps/shim-ts/src/args-hash.ts` -> `packages/ipc-client/src/args-hash.ts`
- `apps/shim-ts/src/jcs.ts` -> `packages/ipc-client/src/jcs.ts`

**What stays in shim-ts:**
- `guard.ts` (the `Guard` class -- shim-specific wrapper logic)
- `openai.ts` (OpenAI adapter -- shim-specific)
- `types.ts` (shim-level user-facing types: `GuardOptions`, `DecisionResult`, etc.)
- `index.ts` (re-exports everything; continues to re-export from `@loopstorm/ipc-client`)

**Package name:** `@loopstorm/ipc-client`
**License:** MIT (code was already MIT in `apps/shim-ts`)
**Location:** `packages/ipc-client/` (under `packages/` for shared library convention)

**Consumers:**
- `apps/shim-ts` -- import from `@loopstorm/ipc-client` instead of local
- `apps/mcp-proxy` -- import from `@loopstorm/ipc-client`
- `apps/supervisor` -- communicates via HTTP/tRPC, NOT IPC directly, but
  MAY use protocol types for consistency

**Rationale:** The 478 lines being extracted represent a clean, stable
interface (UDS NDJSON client, protocol types, args-hash, JCS, error
types). Both MCP proxy and future consumers need exactly this. Duplicating
it creates maintenance burden; importing from shim-ts couples consumers
to the shim package. Extraction is the smallest credible design.

**Risk:** Low. All existing shim-ts tests continue to work because
shim-ts will import from `@loopstorm/ipc-client` and re-export. The
public API of `@loopstorm/shim-ts` does not change.

**Phase blocking:** Phase 2 (MCP Proxy). Phase 1 (BT) is unaffected.
Can be done in parallel with Phase 1.

### Gate 2: Supervisor System Prompt Design

**Decision:** Define a structured system prompt framework with three
sections: ROLE, CONSTRAINTS, and BEHAVIORAL GUIDELINES. Ship an initial
prompt as a TypeScript template string in
`apps/supervisor/src/prompt.ts`. The prompt is a deployment artifact, not
a specification artifact -- it can be iterated without a spec amendment.

**System prompt structure (v1.1 initial):**

```
ROLE:
You are the LoopStorm AI Supervisor. You analyze completed and in-progress
agent runs for risk, anomalies, and policy calibration opportunities.
You operate on the OBSERVATION PLANE ONLY. You cannot modify policies,
budgets, or enforcement rules directly. You create proposals that require
human approval.

CONSTRAINTS:
1. You MUST NOT attempt to execute, modify, or delete any customer data.
2. You MUST escalate to a human when you observe critical risk (score >= 76)
   or when you are uncertain about the appropriate response.
3. You MUST provide a confidence score (0.0-1.0) with every proposal. Be
   honest about uncertainty. A low confidence score with good reasoning is
   better than a high confidence score with shallow analysis.
4. You MUST cite specific run_ids and event sequences as evidence for any
   claim.
5. You are budget-constrained. Use observation tools efficiently: request
   specific event types, use offset_seq for pagination, limit to relevant
   events.

BEHAVIORAL GUIDELINES:
1. Start every session by reading the triggering run's events with
   read_run_events.
2. Compute a risk score before deciding on further action.
3. For LOW risk (0-25): record the pattern and exit. Do not escalate.
4. For MEDIUM risk (26-50): analyze deeper, record findings, flag for review
   if pattern is recurring.
5. For HIGH risk (51-75): analyze, propose specific corrective action,
   escalate if the run is still active.
6. For CRITICAL risk (76-100): escalate immediately, then analyze.
7. When proposing budget adjustments, always compare against the agent's
   baseline to justify the change.
8. When in doubt, escalate. False positives are acceptable; missed critical
   issues are not.
```

**Rationale:** The spec (Appendix E) explicitly states the system prompt
is out of scope. However, leaving this as a fully open question blocks
implementation. The prompt above is a reasonable starting point that:
- Respects enforcement/observation plane separation
- Establishes escalation thresholds tied to the risk score tiers
- Emphasizes evidence-based reasoning
- Encourages conservative behavior (escalate when uncertain)
- Is short enough to leave room for tool schemas in the context window

**Override mechanism:** Mode 1 operators can replace the system prompt
entirely via a `LOOPSTORM_SUPERVISOR_SYSTEM_PROMPT` environment variable
or a `system_prompt_path` configuration field. Mode 2/3 uses the built-in
prompt.

**Phase blocking:** Phase 4 (SUP-B). Not needed for Phase 1, 2, or 3.

### Gate 3: Trigger Queue Mechanism

**Decision:** In-process queue using a bounded async channel within the
backend process. No external queue (Redis, SQS, etc.) in v1.1.

**Implementation:**
- Use a bounded channel (capacity: 100) in the backend process.
- The event ingest handler evaluates trigger conditions synchronously.
- If a trigger fires, it enqueues a `TriggerMessage` into the channel.
- A background worker in the backend process reads from the channel and
  dispatches supervisor sessions via HTTP to the supervisor process.
- If the channel is full (100 pending triggers), new triggers are dropped
  with a warning log. This is acceptable because trigger deduplication
  (60-second window) already limits the rate.

**TriggerMessage shape:**
```typescript
interface TriggerMessage {
  trigger: string;       // "terminated_budget" | "loop_detected" | etc.
  trigger_run_id: string; // The agent run that triggered
  tenant_id: string;
  priority: number;       // 1=critical, 2=high, 3=medium, 4=low
  enqueued_at: string;    // ISO 8601
}
```

**Rationale:**
- An external queue (Redis) adds an infrastructure dependency that
  violates Mode 1 simplicity. The backend process is single-instance in
  v1.1 (no horizontal scaling). An in-process channel is sufficient.
- The supervisor processes triggers asynchronously (seconds-to-minutes
  latency is acceptable). There is no need for persistent queuing or
  guaranteed delivery -- if the backend restarts, pending triggers are
  lost, but the same conditions will re-trigger on the next relevant
  event.
- v2 MAY replace this with a persistent queue (PostgreSQL `LISTEN/NOTIFY`
  or Redis) when horizontal scaling is introduced.

**Phase blocking:** Phase 4 (SUP-A6). Not needed for Phase 1, 2, or 3.

### Gate 4: LLM Provider Abstraction

**Decision:** Thin wrapper interface with a concrete Anthropic SDK
implementation. NOT a full provider-agnostic abstraction in v1.1.

**Interface:**
```typescript
// apps/supervisor/src/llm/provider.ts
interface LLMProvider {
  chat(params: {
    model: string;
    system: string;
    messages: Message[];
    tools: ToolDefinition[];
    max_tokens: number;
  }): Promise<LLMResponse>;
}

interface LLMResponse {
  content: ContentBlock[];
  stop_reason: "end_turn" | "tool_use" | "max_tokens";
  usage: { input_tokens: number; output_tokens: number };
}
```

**Concrete implementation:**
```typescript
// apps/supervisor/src/llm/anthropic.ts
class AnthropicProvider implements LLMProvider {
  constructor(apiKey: string, options?: { baseUrl?: string });
  async chat(params): Promise<LLMResponse>;
}
```

**Rationale:**
- v1.1 targets Claude (haiku-class) only. A full multi-provider
  abstraction (OpenAI, Gemini, Ollama, etc.) is overengineering for v1.1.
- The `LLMProvider` interface is narrow enough that adding a second
  provider in v1.2 or v2 requires implementing one method.
- The interface normalizes the response shape, so the session manager
  never imports `@anthropic-ai/sdk` directly -- it only depends on
  `LLMProvider`.
- Mode 1 operators who want a different provider can implement the
  `LLMProvider` interface. The interface is exported from
  `apps/supervisor/` for this purpose.

**Licensing note:** `@anthropic-ai/sdk` is MIT. It lives in
`apps/supervisor/` (MIT). The `LLMProvider` interface has no SDK
dependency. Only `anthropic.ts` imports the SDK.

**Phase blocking:** Phase 4 (SUP-B2). Not needed for Phase 1, 2, or 3.

### Gate 5: Risk Score Algorithm Weights

**Decision:** Define a fixed-weight linear scoring model with 8 signals.
Weights are compile-time constants (not operator-configurable in v1.1).

**Signals and weights:**

| Signal | Weight | Condition | Max contribution |
|---|---|---|---|
| `high_deny_rate` | 25 | deny_count / total_decisions > 2x agent baseline | 25 |
| `budget_near_cap` | 20 | cost_usd > 0.80 * hard_cap | 20 |
| `budget_exceeded` | 30 | run terminated by budget | 30 |
| `loop_detected` | 20 | any loop_detected event in run | 20 |
| `loop_unrecovered` | 15 | loop detected AND no behavior change after cooldown | 15 |
| `rapid_fire` | 10 | median inter_call_ms < 100 for the run | 10 |
| `structural_drift` | 10 | >3 distinct param_shape_hash values for the same tool | 10 |
| `abnormal_token_spike` | 15 | any token_rate_delta > 5.0 in the run | 15 |

**Scoring algorithm:**
```
raw_score = SUM(signal.weight for each signal where condition is true)
score = MIN(raw_score, 100)  // cap at 100
```

**Tier mapping (from spec Section 4.2):**
- 0-25: LOW
- 26-50: MEDIUM
- 51-75: HIGH
- 76-100: CRITICAL

**Confidence calculation:**
```
confidence = MIN(1.0, event_count / 10) * baseline_factor
```
Where `baseline_factor` is:
- 1.0 if the agent has >= 10 historical runs
- 0.7 if the agent has 3-9 historical runs
- 0.4 if the agent has < 3 historical runs

**Rationale:**
- Linear weighted scoring is interpretable and debuggable. The supervisor
  can explain exactly which signals contributed to the score and by how
  much.
- Weights are chosen so that a single critical signal (budget_exceeded at
  30) plus any two secondary signals push the score into HIGH territory.
  Two critical signals plus secondaries push into CRITICAL.
- The max possible score is 145 (all signals fire), capped at 100. This
  means CRITICAL is reachable but not trivially.
- Weights can be tuned in v1.2 based on observed approval/rejection
  patterns. The v1.1 weights are a calibrated starting point, not final.

**Phase blocking:** Phase 4 (SUP-B4, `compute_risk_score`). Not needed
for Phase 1, 2, or 3.

### Gate 6: OTel Event Store Reader Scope

**Decision:** DEFER event store reader to v1.2. The v1.1 OTel exporter
ships with the JSONL file reader only.

**Rationale:**
- The JSONL reader is the Mode 0 priority. Mode 0 is the primary
  distribution mode.
- The event store reader requires a PostgreSQL connection string and
  database access. This is a Mode 2/3 feature.
- The v1.1 OTel exporter already has enough work: JSONL tail-follow,
  span building, ID derivation, buffering, all 12 event types, BT field
  mapping, supervisor span links. Adding a database polling reader risks
  scope creep.
- The exporter architecture cleanly separates input sources (reader) from
  span building (builder). Adding the event store reader in v1.2 is a
  localized change.

**What ships in v1.1:**
- `apps/otel-exporter/src/jsonl-reader.ts` -- full JSONL tail-follow
- `apps/otel-exporter/src/span-builder.ts` -- full span building
- `apps/otel-exporter/src/ids.ts` -- ID derivation
- `apps/otel-exporter/src/config.ts` -- configuration

**What is deferred to v1.2:**
- `apps/otel-exporter/src/eventstore-reader.ts` -- PostgreSQL polling

**Spec impact:** The OTel spec (Section 8.2.2) already describes the
event store reader as a separate input source. No spec change needed.
The v1.1 implementation plan's OTEL-5 step is deferred.

**Phase blocking:** None. The JSONL reader is sufficient for v1.1
acceptance criteria.

### Gate 7: R7 -- `escalate_to_human` Rule ID Reconciliation

**Decision:** Update the engine to use
`__builtin_escalate_to_human_allow`. The spec's value is canonical.

**Rationale:**
- The spec value `__builtin_escalate_to_human_allow` is more descriptive:
  it encodes both the tool name AND the action (allow) in the rule_id.
  This follows the pattern of other builtin rule_ids which encode their
  behavior (e.g., `__builtin_ipc_parse_error` encodes the error type).
- The engine code is the implementation; the spec is the contract.
  Changing the implementation to match the contract is the correct
  direction.
- The v1.0 release used `__builtin_escalate_to_human`. Changing it in
  v1.1 is a minor breaking change for consumers that pattern-match on
  rule_id. This is acceptable because:
  - The v1.1 release notes will document it.
  - No public API consumer should hardcode builtin rule_ids.
  - The only hardcoded references are in tests (which we update).

**Files to update:**
1. `apps/engine/src/evaluator.rs` -- line 46: change rule_id string
2. `apps/engine/src/evaluator.rs` -- test assertions (3 locations)
3. `apps/engine/tests/ipc_integration.rs` -- line 197: test assertion
4. `apps/engine/src/lib.rs` -- test (if it asserts rule_id)

**Phase blocking:** Phase 1 (BT). The engine changes in Phase 1 should
include this fix so that all v1.1 engine work ships with the correct
rule_id.

## Consequences

### Positive

1. All seven gates are resolved. Implementation can begin on all four
   phases without architectural ambiguity.
2. The IPC client extraction creates a clean, reusable package that
   benefits both MCP proxy and future consumers.
3. The risk score algorithm is deterministic and interpretable, matching
   the product requirement that interpretation tools use no LLM.
4. The in-process trigger queue avoids adding infrastructure dependencies.
5. The LLM provider interface is narrow enough to support future
   providers without premature abstraction.

### Negative

1. The rule_id change (`__builtin_escalate_to_human` ->
   `__builtin_escalate_to_human_allow`) is a minor breaking change for
   v1.0 consumers who pattern-match on the rule_id string.
2. The event store reader deferral means Mode 2/3 users cannot use the
   OTel exporter with the event store until v1.2.
3. The system prompt is a best-effort starting point. It will need
   iteration based on real supervisor sessions.
4. Risk score weights are calibrated estimates, not empirically derived.
   They will need tuning after v1.1 ships.

### Neutral

1. The `LLMProvider` interface is narrow by design. It does not support
   streaming, vision, or structured outputs. These can be added to the
   interface in v1.2 if needed.
2. The trigger queue capacity (100) may need adjustment under high load.
   Monitoring the drop rate will inform v1.2 tuning.

## Migration Path

### From v1.0 to v1.1

1. **rule_id change:** Update any code that pattern-matches on
   `__builtin_escalate_to_human` to use `__builtin_escalate_to_human_allow`.
   Document in release notes.
2. **New packages:** `packages/ipc-client/` is new. `apps/shim-ts` now
   depends on it. Consumers of `@loopstorm/shim-ts` see no API change.
3. **Event schema:** 4 new optional fields on `policy_decision` events.
   No schema_version bump. Existing consumers ignore them.
4. **CLI compatibility:** v1.0 CLI cannot verify v1.1 JSONL files (BT
   fields change the hash). CLI must be updated to v1.1 alongside the
   engine.

### From v1.1 to v1.2

1. **OTel event store reader** added. No breaking changes.
2. **Risk score weights** may be adjusted based on empirical data. The
   interface does not change.
3. **System prompt** may be revised. The interface does not change.
4. **Trigger queue** may be replaced with a persistent queue if
   horizontal scaling is introduced. The `TriggerMessage` shape is
   stable.
