# LoopStorm Guard v1.1 — Production Readiness Audit

**Date:** 2026-04-07
**Auditor:** Lead Architect
**Commit:** 919dcb8 (main)
**Verdict:** NOT READY for commercial use. Ready for OSS Mode 0 only.

---

## Executive Summary

LoopStorm Guard has impressive engineering depth: a real Rust enforcement engine
with 80+ unit tests, a working Python shim, TypeScript shim, CLI, full tRPC
API surface, a Next.js dashboard with a polished landing page, an AI supervisor,
MCP proxy, and OTel exporter.

**However, it has never been run as a connected system.** No human has ever:

1. Started Supabase locally, run migrations, started the backend, started the
   frontend, signed up, created an API key, run an agent through the engine,
   ingested events, and seen them appear in the dashboard.
2. Deployed the backend to Cloudflare Workers (no wrangler.toml exists).
3. Deployed the frontend to Vercel (no vercel.json, no project configured).
4. Processed a payment.
5. Sent an email (email verification is required but no SMTP is configured).

The product is a collection of individually well-built components that have
never been wired together end-to-end in a real environment. It is a prototype
at the integration layer, despite being production-grade at the module layer.

**Current state:** Better Auth is configured with email+password and optional
Google OAuth. Session management uses cookie-based caching with 5-minute TTL.
Tenant provisioning hook fires on user creation. Self-healing tenant resolution
exists for edge cases.

**Gaps:**
| Gap | Priority | Effort |
|-----|----------|--------|
| **Email verification is enabled but emails cannot be sent.** No `sendEmail` adapter is configured in the Better Auth options. Registration will succeed but users will be locked out because they cannot verify their email address. This is the single biggest blocker for any real signup. | P0 | 4-8 hours |
| No password reset flow tested end-to-end. Better Auth supports it, but the frontend has no "Forgot Password" page. | P1 | 4 hours |

### 3B. API Key System
**Current state:** Well-designed. SHA-256 hashed storage, timing-safe comparison,
prefix-based lookup, expiry support, revocation, fire-and-forget last_used_at
updates. Create/list/revoke tRPC procedures exist. Frontend UI exists for
creating and managing keys.

**Gaps:**
| Gap | Priority | Effort |
|-----|----------|--------|
| API key scopes are stored but never enforced. The `apiKeyProcedure` middleware exposes `apiKeyScopes` on context but no procedure checks them. A key with `["read"]` scope can call `events.ingest`. | P1 | 4 hours |
| No key rotation support. Users must create a new key and revoke the old one. | P3 | 4 hours |
| The seed API key in seed.sql has scopes `['ingest', 'read']` but there's no concept of a "supervisor" scope for the supervisor's API key -- it just uses any valid key. | P2 | 2 hours |

**CORRECTION:** The supervisor-tools router DOES check scopes. The
`apiKeyProcedure` context exposes scopes, and supervisor-tools checks for
`"supervisor"` scope. The seed key does not include this scope, which means
creating a supervisor-scoped key requires going through the dashboard or
a direct DB insert. This is correct behavior but needs documentation.

## 8. Security

### 8A. What's Done Well
- **Fail-closed enforcement** (ADR-002) -- verified in engine + tests
- **Hash-chain audit trail** -- SHA-256 chain, CLI-verifiable
- **`escalate_to_human` invariant** -- 5 enforcement points verified
- **Secret redaction** -- API keys, JWTs, AWS creds auto-redacted from logs
- **Socket permissions** -- 0600 on UDS
- **RLS** -- 6 tables with tenant isolation + FORCE ROW LEVEL SECURITY
- **Timing-safe API key comparison** -- crypto.timingSafeEqual
- **API key hashing** -- SHA-256, raw key never stored
- **CORS** -- configured, strict in production
- **SPDX headers** -- CI-enforced
- **Secret scanning** -- gitleaks configured
- **Dependency license audit** -- cargo-deny + JS license check

### 8B. Security Gaps
| Gap | Priority | Effort |
|-----|----------|--------|
| **RLS context may leak between requests.** Non-transactional handlers (runs.list, policies.list, etc.) call `setTenantRlsContext` but don't wrap their queries in a transaction. With connection pooling, the `LOCAL` scoping may not apply (set_config with `true` = LOCAL to transaction, but without a transaction, it's local to the session/connection). If two requests share a pooled connection sequentially, the second inherits the first's RLS context until it sets its own. | P0 | 1-2 days |
| **No rate limiting.** The API has no rate limiting on any endpoint. A single client can hammer the ingest endpoint, auth endpoints, or tRPC procedures without throttling. | P0 (for production) | 1-2 days |
| **No CSRF protection.** Better Auth cookies with CORS credentials should be sufficient, but explicit CSRF tokens are not used. | P2 | 4 hours |
| **No request body size limits.** The Hono server does not configure a maximum request body size. The tRPC ingest endpoint limits to 1000 lines but doesn't limit total payload size in bytes. | P1 | 2 hours |
| **API key scope enforcement is partial.** See 3B. | P1 | 4 hours |
| **No audit log for admin actions.** Dashboard actions (approve proposal, revoke key, create policy) are not logged. Only engine enforcement events are logged. | P2 | 2-3 days |
| **`SUPABASE_SERVICE_ROLE_KEY` is required but unused.** This is a key that bypasses RLS. Requiring it in the environment when it's never used creates risk -- if it accidentally gets referenced, it would bypass all tenant isolation. Remove it from env.ts. | P1 | 30 min |
| Session tokens are stored in plain text in the sessions table. Better Auth handles this, and it's industry-standard (HttpOnly cookies), but worth noting. | P3 | N/A (framework standard) |

## 9. Missing Integration Points

### 9A. End-to-End Data Flow (NEVER TESTED)

The intended flow is:
```
Agent -> Python Shim -> Engine (UDS) -> JSONL file
                                            |
                                            v (HTTP batch sink)
                                        Backend API (events.ingest)
                                            |
                                            v
                                        PostgreSQL
                                            |
                                            v
                                        Web Dashboard
```

**What has been tested:**
- Agent -> Shim -> Engine -> JSONL: YES (Mode 0 smoke test, 88 Python tests)
- Engine -> JSONL file: YES (integration tests)
- Backend events.ingest with raw JSONL lines: YES (unit tests with mocked DB)
- Frontend rendering pages: YES (Playwright tests, but against mocked data)

**What has NEVER been tested:**
- Shim -> Engine -> JSONL -> HTTP sink -> Backend ingest: NO
- Backend -> PostgreSQL with real Supabase: NO (tests use BUN_ENV=test stubs)
- Frontend -> Backend -> PostgreSQL full stack: NO
- Sign up -> verify email -> login -> create key -> ingest events -> view runs: NO
- Supervisor receive trigger -> run session -> create proposal -> dashboard shows it: NO

### 9B. HTTP Sink (Engine -> Backend)
**The HTTP sink does not exist in the engine.** The engine writes JSONL to a local
file. There is no code anywhere in the engine or shims that forwards events to the
backend API. The entire Mode 2 data flow (local engine + hosted backend) depends on
a component that has not been built.

**Options that exist but are disconnected:**
1. The Python shim and TS shim could theoretically send events to the backend,
   but they don't have this capability either.
2. The `loopstorm replay` CLI could theoretically be extended to pipe events to
   the backend, but it doesn't.
3. A separate "forwarder" process could tail the JSONL file and POST to the
   ingest endpoint, but this doesn't exist.

This is a fundamental gap in the Mode 2 architecture. **There is no path from
the local engine to the hosted backend.**

| Gap | Priority | Effort |
|-----|----------|--------|
| **No HTTP event forwarder exists.** The Mode 2 data flow is broken. | P0 (for Mode 2) | 1-2 weeks |
| No batch upload command in the CLI (e.g., `loopstorm upload audit.jsonl --api-key lsg_xxx --url https://api.loop-storm.com`). | P0 (for Mode 2) | 2-3 days |

### 9C. tRPC Frontend-Backend Integration
**The frontend and backend type-share correctly** -- the frontend imports
`AppRouter` type from `@loopstorm/api`. However, the frontend has never been
tested against a real backend. All Playwright tests either mock the backend or
require the full stack to be running (and the Playwright CI job is deferred).

## 11. Honest Verdict

**The engineering is real and solid.** The Rust engine is production-grade. The
architectural decisions (enforcement/observation separation, fail-closed, hash
chains, RLS) are genuinely thoughtful. The test coverage is strong at the unit
level. The code quality is high.

**But this is not a product yet.** It is a collection of well-engineered
components that have never been assembled into a running system. The gaps are
not in the code quality -- they are in the integration, deployment, and business
infrastructure layers.

The most critical false signal is that the deploy.yml and release.yml exist and
look correct, but neither has ever produced a successful deployment. The entire
hosted product (Mode 2/3) depends on infrastructure that does not exist.

**Recommended path:**

1. **Week 1:** Fix Mode 0 blockers. Publish packages. Exercise release pipeline.
   Write local dev guide. Get the domain up.

2. **Week 2-3:** Fix the backend deployment story. The Cloudflare Workers
   assumption needs to be re-evaluated -- the backend uses `setInterval`
   background jobs which Workers cannot run. Consider Fly.io, Railway, or
   a plain VPS with Bun. Set up Supabase production. Fix email verification.
   Fix RLS scoping.

3. **Week 4-6:** Build the event forwarder (the Mode 2 missing link). Run the
   full stack end-to-end. Add rate limiting. Add request body limits.

4. **Week 7-14:** Stripe integration, legal docs, documentation site, support
   channels.

Do not release Mode 2/3 until someone has personally signed up, verified their
email, created an API key, run an agent with the engine, had events appear in
the dashboard, approved a supervisor proposal, and verified a hash chain --
all in a single session on production infrastructure.
